# LiveNode Decision Trace — 設計書（日本語版）

## 目的

まとまりのない会議メモや検討文を入力すると、人間が追跡できる「判断の構造」に変換する、OpenAI Build Week向けの公開デモを作る。

審査員が説明を読まなくても、30秒以内に次の操作を完了できることを目標とする。

1. サンプルを選ぶ
2. Decision Traceを生成する
3. 推奨案とその理由を理解する
4. 結果をMarkdownでコピーする

これは、LiveNodeの中核思想である「AIは答えだけでなく、そこへ至った判断プロセスも残すべきだ」を体験できるプロダクトとする。

## 今回作るもの

- 1ページで完結する公開Webアプリ
- 自由入力できる文章欄
- ワンクリックで使える3種類のサンプル文章
- OpenAIによる構造化分析
- 6項目からなるDecision Trace
- 入力文章のどこを根拠にしたかを示す短い引用
- Markdown形式でのコピー
- 日本語・英語入力への対応
- PC・スマートフォン両対応
- アカウント・データベースなしの一時処理

## 今回は作らないもの

- PDFやファイルのアップロード
- ユーザー登録やログイン
- クラウド上の履歴保存
- チーム共同作業
- 複数AIエージェントによる討議
- 個人のAI-Brainへの接続
- 生成後にカードを個別編集する機能

期限内に完成度を高めるため、判断構造を生成する中心体験だけに絞る。

## 利用の流れ

1. 利用者がページを開き、タイトルと短い説明から用途を理解する。
2. 自分の検討文章を貼り付けるか、3つのサンプルから1つ選ぶ。
3. **Generate Decision Trace** ボタンを押す。
4. サーバーが入力を確認し、OpenAIへ構造化分析を依頼する。
5. 推奨案の概要と6枚のカードが順番に表示される。
6. 結果全体をMarkdownでコピーするか、最初からやり直す。

サンプルは次の3分野を用意する。

- 商品・事業上の判断
- 自治体・公共政策上の判断
- 日常業務の運営判断

どのサンプルも答えが最初から明白なものではなく、複数の選択肢に本当の利点と代償がある内容にする。

## Decision Traceの6項目

### 1. Situation — 状況

何を決めなければならないのか、どのような背景があるのかを整理する。

### 2. Assumptions — 前提

文章中に明記された前提と、判断に影響している隠れた前提を分けて示す。

### 3. Decision criteria — 判断軸

何を大切にして決めるべきか、制約条件や成功基準を整理する。

### 4. Options and trade-offs — 選択肢と代償

現実的な選択肢ごとに、利点・費用・リスク・後から変更できるかを示す。

### 5. Recommendation — 推奨案

推奨する選択肢、その理由、確信度、どの条件が変われば結論も変わるかを示す。

### 6. Next actions — 次の行動

不確実性を減らしたり、判断を実行へ移したりするための具体的な行動を順番に示す。

分析上の主張には、可能な限り入力文章から短い根拠を付ける。入力された事実とAIによる推論を明確に分け、与えられていない事実を作らない。

## 画面設計

画面は「入力」「生成中」「結果」の3状態だけにする。

### 入力画面

- プロダクト名と一文説明
- 大きな文章入力欄
- 3つのサンプル選択ボタン
- 文字数と簡単な入力案内
- 生成ボタン

### 生成中

- 入力した文章を消さず、そのまま画面に残す
- Decision Traceの各段階に対応した進行表示を出す
- 処理中の二重送信を防ぐ

### 結果画面

- 最上部に推奨案と確信度を表示
- 6項目を順番にカード表示
- 入力文からの根拠とAIの推論をラベルで区別
- **Markdownをコピー** と **最初からやり直す** ボタン

LiveNodeらしい宇宙・星座の雰囲気を薄く取り入れる。ただし、装飾よりも30秒で内容を理解できる読みやすさを優先する。

## 技術構成

- アプリ：Next.js＋TypeScript
- デザイン：Tailwind CSS
- AI：OpenAI Responses API＋構造化出力
- 公開：Cloudflare Pages／Workers対応環境
- データ保存：なし。ページを閉じるかリセットすると消える
- APIキー：サーバー側の環境変数にだけ保存
- AIモデル：環境変数で変更可能にし、大会Resourcesで推奨される対象モデルを採用

ブラウザから同じドメイン内のサーバーAPIへ送るのは、入力文章と使用言語だけとする。サーバー側で文字数を確認し、OpenAIへ送信し、決められたJSON形式になっているか検証してから画面へ返す。

Markdownへの変換はブラウザ側で行う。

## 入力不備・エラーへの対応

- 空欄や極端に短い文章は、OpenAIへ送る前に不足情報を案内する。
- 何を決めたい文章なのか判別できない場合は、「決めたいことを一文で追加してください」と案内する。
- 通信やAPIに失敗しても、入力文章を消さず再試行できるようにする。
- AIの返答が決めた形式にならなかった場合、サーバー側で1回だけ自動再試行する。
- それでも失敗した場合、技術的なエラー文を見せず、簡単な言葉で再試行を案内する。
- 公開デモの乱用を防ぐため、入力サイズと利用回数に上限を設ける。
- 医療・法律・金融など重大な判断では、AIが最終決定を代行せず、専門家による確認を案内する。

## プライバシーと安全性

- 入力文章を保存するデータベースを作らない。
- 入力文章そのものをアクセス解析やアプリログへ記録しない。
- OpenAIのAPIキーや詳しいエラー情報をブラウザへ出さない。
- AIの出力を未加工のHTMLとして表示しない。
- 入力文章は信用できないデータとして扱い、AIへのシステム指示と分離する。
- 入力付近に「処理のためOpenAIへ送信するが、このアプリでは保存しない」と明記する。

## テスト範囲

### 自動テスト

- 6項目が揃った正常な結果を受け入れ、不完全な結果を拒否できること
- 入力チェックとMarkdown変換が正しく動くこと
- API成功・失敗・時間切れ・形式不正を処理できること
- ブラウザへ送られるファイルにAPIキーが含まれないこと

### 本番環境での実測

- 公開URLで3つのサンプルをすべて実行する
- 短文・長文・日本語・英語を試す
- PC幅とスマートフォン幅で一連の操作を確認する
- 失敗時に入力が残り、再試行できることを確認する
- Markdownコピーに6項目すべてが含まれることを確認する
- 通常時にサンプル選択から結果表示まで30秒以内であることを測る

## Devpost提出に必要な成果物

- 誰でも試せる公開URL
- 公開GitHubリポジトリ、または審査可能なコードリンク
- サンプル選択、生成、根拠表示、Markdownコピーを見せる60〜90秒のデモ動画
- 3:2比率のサムネイル
- 画面スクリーンショット2〜3枚
- 開発の動機・実装方法・学び・苦労をまとめたプロジェクト説明

## 完成条件

- 初めて訪れた人が説明なしで中心操作を完了できる。
- 生成に成功した結果には、必ず6項目が含まれる。
- 推奨案が入力文章の根拠に結び付いており、AIの推論と区別されている。
- 入力文章をアプリ側で保存しない。
- APIキーがブラウザへ漏れない。
- PCとスマートフォンの本番環境で一連の操作を確認済みである。
- 完成したアプリからDevpostに必要なURL・動画・画像・説明文を用意できる。
