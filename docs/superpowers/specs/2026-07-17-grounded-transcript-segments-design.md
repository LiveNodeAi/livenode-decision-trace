# 長文文字起こしの根拠区間検出 仕様

## 目的

Circleback等の長い文字起こしに改行・反復表現・同一文が含まれても、テーマ検出後に確認画面へ確実に進めるようにする。

## 現在の問題

現在はAIに原文引用を返させ、サーバーが引用を完全一致かつ一意一致で検索している。AIが空白や改行をわずかに正規化した場合、または同じ表現が複数回ある場合、内容理解が正しくても`MALFORMED_RESPONSE`になる。

## 採用方式

### 1. サーバーで原文を区間化

- 文字起こしを原文のUTF-16位置を保持した連続区間へ分割する。
- 改行、文末記号、最大長を境界候補にする。
- 各区間へ`segment-1`形式の安定ID、`start`、`end`、原文`text`を付ける。
- 空白や改行を変更せず、`transcript.slice(start, end) === text`を必須にする。
- 1区間は最大800 UTF-16 code unit、全区間で原文30,000文字を欠落なく扱う。

### 2. AIは区間IDを選ぶ

- AIへは番号付き区間の配列を非信頼データとして渡す。
- テーマごとに1〜6個の`segmentIds`を返させる。
- AIに引用文、開始位置、終了位置を生成させない。
- 最大5テーマ、低料金モデル`gpt-5.4-nano`、reasoning none、最大出力2,500を維持する。

### 3. サーバーで根拠範囲へ復元

- 返されたIDが存在し、テーマ内・テーマ間で重複していないことを検証する。
- IDから`start`、`end`、`excerpt=text`へ決定論的に変換する。
- 変換後は既存の`validateSourceRanges`を必ず通す。
- 存在しないID、7区間、6テーマ、重複ID、範囲改変は拒否する。

## エラーと再試行

- provider応答が不正な場合だけ既存どおり最大1回再試行する。
- 公開エラーは本文やprovider詳細を含めない。
- UIの失敗表示は、テーマ検出失敗であることが分かる日本語へ変更する。
- 入力内容を保持し、利用者が再入力しなくてよい状態を維持する。

## 互換性

- 公開APIの成功レスポンス`TopicDetection`は変更しない。
- 後段のテーマ確認、最大2並列分析、Decision Trace、ZIPは変更しない。
- 単一のアイデアメモ機能は変更しない。

## テスト

- Circleback風の改行・短い発話・反復表現を含む6,595文字相当をモックで検出できる。
- 絵文字を含む区間のUTF-16位置が完全一致する。
- 同じ文章が複数回あっても異なる区間IDとして区別できる。
- 逆順の区間IDを原文順へ並べ直す。
- 存在しないID、重複利用、7区間、6テーマを拒否する。
- 全既存テスト、PC幅・375px E2E、本番ビルドを通す。
- 実API検証は再デプロイ後にテーマ検出1回だけ行い、成功時だけ2テーマ分析を各1回行う。
