# LiveNode Decision Trace — 複数テーマ文字起こし・ZIP出力 設計仕様書

作成日：2026-07-17
状態：設計承認済み・実装計画作成前

## 1. 目的

現在の「1つの判断メモから1つのDecision Traceを作る公開デモ」を、会議の文字起こしやAIセッションログを丸ごと扱える体験へ拡張する。

利用者は長い文字起こしを貼り付け、AIが検出した最大5つの判断テーマを確認し、必要なテーマだけを選び、テーマごとにDecision Traceを生成する。完成した結果は、会議サマリー・テーマ別Decision Trace・KX Note・アクション一覧を含むMarkdown ZIPとして一括保存できる。

このWebアプリは、より大きなKX構想の「取り込み体験」を公開するものと位置づける。

- **Web体験版**：会話を取り込み、テーマへ分割し、根拠を確認し、判断過程を可視化・出力する
- **ローカルKX**：現在すでに運用している、人間の確認を含むMarkdown蒸留・精錬パイプライン
- **配布Skill版**：今後、一般利用者へ届ける実行・保存レイヤー

公開デモでは、Obsidian・Notionへの自動保存、異なるAIへの自動導入、長期的な人格再現がすでに完成しているとは説明しない。

## 2. 中心思想

一般的なAIメモリーは、事実、要約、最終回答を保存する。LiveNodeが残そうとしているのは、人間の判断プロセスである。

- 何を主張として考えたか
- 何を根拠として信じたか
- どんな制約を受け入れたか
- どの選択肢を退けたか
- 何との接続によって判断したか

この構造は、汎用LLMと利用者本人の間に置かれる中間レイヤーになる。AIに人間を完全再現させるものではない。未来のAI作業を、全人類的な高水準の平均だけでなく、その人自身の経験・価値観・判断軸へ寄せるための記憶である。

同時に、人間自身が過去の結論だけでなく、「なぜそう考えたのか」を振り返るための記憶でもある。

中心メッセージ：

> 会話を、単なる決定事項ではなく「なぜそう判断したのか」を再利用できる記憶へ変える。

英語の短い説明：

> LiveNode Decision Trace turns a conversation into a reusable memory of how a decision was made—not just what was decided.

## 3. 今回の実装範囲

### 実装するもの

- 日本語・英語の文字起こしを最大30,000文字まで入力
- 最大5つの判断テーマを検出
- 生成前にテーマを人間が確認
- テーマの選択・除外・タイトル修正
- 選択テーマごとに1つのDecision Traceを生成
- 同時生成は最大2テーマ
- 一部テーマが失敗しても、成功した結果を保持
- 失敗テーマだけを個別に再試行
- 成功したTraceから会議サマリーとアクション一覧を決定論的に生成
- Markdown ZIPをブラウザ側で生成・ダウンロード
- Web版・ローカルKX・将来Skill版の関係を画面と応募文で説明

### 今回は実装しないもの

- WebアプリからObsidian、Notion、Google Drive、AI-Brainへ直接保存
- OAuth認証やユーザーアカウント
- 文字起こし、テーマ、生成結果、ZIPのサーバー保存
- 6テーマ以上の生成
- 検出された元発言範囲の編集
- 完成したテーマ別Traceを超える、AIによる会議全体の追加解釈

## 4. 利用の流れ

### ステップ1：文字起こしを丸ごと貼る

入力欄には、会議の文字起こし、Circleback等の記録、AIとのセッションログを貼り付けられる。

利用者が「判断テーマを検出」を押す前に、次を表示する。

- 最大30,000文字
- 公開デモでは最大5テーマ
- 当アプリは入力を保存しない
- 処理のためOpenAIへ送信する

現在の3つの短いサンプルは、すぐ試せる導線として残す。

### ステップ2：分割結果を確認する

検出されたテーマをカードで表示する。各カードには次を含める。

- 選択チェック
- 修正可能なタイトル
- 短い要約
- 元発言の根拠抜粋
- 判断を含まない会話を除外したことの表示

利用者はテーマの選択・除外とタイトル修正ができる。元発言の範囲は編集できない。1テーマ以上選ばないと生成できない。

最大5テーマという制限は、公開デモの待ち時間とAPI利用量を抑えるためだと説明する。

### ステップ3：選択テーマを生成する

ブラウザから、最大2件のテーマ分析を並列実行する。各テーマは独立した状態を持つ。

- 待機中
- 生成中
- 完了
- 再試行可能な失敗
- 再試行できない失敗

1テーマが失敗しても、完成した他テーマは消さない。再試行は失敗テーマだけに限定する。二重クリックや同一テーマの重複実行を防ぐ。

### ステップ4：確認・保存する

結果画面の先頭に「会議全体の判断マップ」を表示する。

- 成功したテーマ
- 各テーマの推奨
- 判断を変える条件
- 次のアクション
- 失敗したテーマ

その下に、各テーマの既存6項目Decision Traceと2種類のMarkdown出力を表示する。

実際に動く保存操作は「Markdown ZIPをダウンロード」とする。

Obsidian、Notion、AI-Brainは、動かないボタンにはしない。「Skill版の保存アダプター」として説明カードを表示する。

## 5. システム構成

### テーマ検出API

`POST /api/topics/detect`

テーマ検出は、分類・抽出用途に適した`gpt-5.4-nano`を使用し、reasoning effortは`none`とする。Decision Trace本体より安価なモデルへ役割を分離し、長い文字起こしの入口コストを抑える。

入力：

```json
{ "transcript": "文字起こし全文" }
```

成功時の出力：

```json
{
  "transcriptHash": "整形済み文字起こしのSHA-256",
  "topics": [
    {
      "id": "topic-1",
      "title": "テーマ名",
      "summary": "短い要約",
      "ranges": [
        { "start": 120, "end": 256, "excerpt": "元発言" }
      ]
    }
  ]
}
```

`start`と`end`は、JavaScriptのUTF-16コード単位の位置とする。

サーバーは検証済み文字起こしのSHA-256を計算し、すべての範囲で次が成立することを確認する。

```text
transcript.slice(start, end) === excerpt
```

テーマ検出時の位置照合では、NFKC等の曖昧な文字正規化を使わず、元文字列との完全一致を要求する。

テーマIDは1〜5件で重複不可。1テーマあたりの範囲数、抜粋の長さ、引用文字数合計、テーマ間の重複率、全文に対するカバー率に上限を設ける。

複数テーマに本当に関係する発言は共有可能だが、各テーマが全文を重複して引用する出力は拒否する。

### テーマ分析API

`POST /api/topics/analyze`

テーマ別Decision Traceは、現在の単一判断モードと同じ`gpt-5.6-luna`を使用し、reasoning effortは`none`とする。ハッカソンの中心となる判断構造化にはGPT-5.6を残し、単純なテーマ分類だけをnanoモデルへ分担させる。

入力：

```json
{
  "transcript": "文字起こし全文",
  "transcriptHash": "...",
  "topic": {
    "id": "topic-1",
    "editedTitle": "利用者が確認・修正したタイトル",
    "ranges": [
      { "start": 120, "end": 256, "excerpt": "元発言" }
    ]
  }
}
```

サーバーは文字起こしのハッシュと、すべての元発言範囲を再検証する。

修正されたタイトルは、命令ではなく信頼できない入力データとして扱う。タイトル内に指示文が書かれても、システム指示を上書きできない。

OpenAIへは30,000文字全文をテーマ数分送らない。検証済みの元発言範囲と、その前後の限定した文脈だけを送る。これにより、根拠、料金、待ち時間を安定させる。

成功時は、既存の検証済み`DecisionTrace`に次を付けて返す。

- `topicId`
- 検証済み元発言範囲
- `highImpact`
- 外部へ意味を持たない試行ID

既存の根拠サニタイズはテーマごとに適用する。選択したテーマ自体が元発言に根拠を持たない場合は、成功扱いにせず`TOPIC_NOT_GROUNDED`を返す。

### ブラウザ側の並列処理

小さなPromise Poolを使い、同時に動くテーマ分析を最大2件に制限する。

テーマ状態はサーバーが発行したテーマIDごとに独立管理する。1テーマの中止、再試行、失敗が、完成した別テーマを壊してはいけない。

### 保存しない設計

アプリ内には、文字起こし、テーマ、Trace、ZIPを保存するデータベースを置かない。リクエスト本文やモデル出力をログへ出さない。

OpenAIには`store: false`を指定する。ただし画面では「当アプリは入力を保存せず、処理のためOpenAIへ送信する」と正確に書く。すべてのインフラ事業者が一切保持しないとは断言しない。

## 6. 入力制限・料金・不正利用対策

- 文字数上限：30,000文字
- 通信サイズ上限：JSON分を含め最低140KB
- JSON解析前と解析後の両方で上限確認
- テーマ数上限：5
- ブラウザの同時分析：2
- 構造化出力が壊れた場合のAI再試行：1回
- テーマ検出の最大出力：2,500トークン
- 1テーマのDecision Trace最大出力：2,500トークン
- テーマ検出とテーマ分析は別々のCloudflareレート制限キーを使用
- 公開エラーに含めるのは、安定したエラーコード、再試行可否、必要ならテーマIDだけ
- 文字起こし、抜粋、修正タイトルはすべて信頼できないデータとしてプロンプトインジェクション対策を継承
- Reactのテキスト表示とMarkdown生成のみを使用し、raw HTMLは表示しない

### API料金の予算

モデルの役割と2026-07-17時点の公式単価を次のように分ける。

| 処理 | モデル | 入力単価 / 100万トークン | 出力単価 / 100万トークン |
|---|---|---:|---:|
| テーマ検出 | `gpt-5.4-nano` | $0.20 | $1.25 |
| Decision Trace | `gpt-5.6-luna` | $1.00 | $6.00 |

料金目標：

- 短い単一Decision Trace：1セント前後以下を目安
- 30,000文字・5テーマの通常処理：合計$0.10未満を目標
- 構造不良による再試行を含む最悪系：合計$0.20前後を設計上限

これは保証価格ではなく、入力トークン、出力トークン、再試行回数に依存する設計目標である。実装後はOpenAI Usageの実測値を記録し、テーマ数ごとの概算料金を検証資料へ掲載する。

公開デモでは、利用者へ料金を表示する必要はないが、内部では1セッションのプロバイダー呼び出し数と推定トークン上限を監視できる形にする。開発時の本番受入は同じ入力を何度も繰り返さず、モック・ローカルテストを優先し、最後に限定した実リクエストだけを行う。

公開エラーコード：

- `INVALID_REQUEST`
- `TRANSCRIPT_TOO_SHORT`
- `TRANSCRIPT_TOO_LONG`
- `REQUEST_TOO_LARGE`
- `RATE_LIMITED`
- `HASH_MISMATCH`
- `INVALID_SOURCE_RANGE`
- `TOO_MANY_TOPICS`
- `TOPIC_NOT_GROUNDED`
- `PROVIDER_TIMEOUT`
- `PROVIDER_REFUSED`
- `PROVIDER_BUSY`
- `MALFORMED_RESPONSE`

## 7. 会議サマリー

会議サマリーは、成功したテーマ別Traceだけから作る純粋な変換処理とする。追加のAI呼び出しはせず、新しい主張も加えない。

含める内容：

- 利用者が入力した会議名・日付、または中立的なファイル名
- 画面上の順番どおりの選択テーマ
- 各テーマの推奨
- 判断を変える条件を、未決事項・監視事項として表示
- テーマ別にまとめた全アクション
- 失敗テーマの明示
- 対応するMarkdownファイルへのリンク

同じTraceを入力すれば、常にバイト単位で同じサマリーMarkdownを生成する。

## 8. ZIP構造

```text
YYYY-MM-DD_会議名/
├── 00-meeting-summary.md
├── 01-safe-topic-name/
│   ├── Decision-Trace.md
│   └── KX-Note.md
├── 02-safe-topic-name/
│   ├── Decision-Trace.md
│   └── KX-Note.md
├── ...
├── 99-actions.md
└── manifest.json
```

連番を正式な並び順とする。ファイル名は記号、長さ、重複、パストラバーサルを防ぐ形に変換する。日本語タイトルはMarkdown本文に残す。

`manifest.json`には、全文を複製せず、テーマ順、状態、元発言範囲情報、対応ファイル名を記録する。

1テーマでも成功していればZIPを作成できる。全テーマ失敗時はZIPを作成できない。

## 9. 画面・応募時の説明

画面では3層を明確に説明する。

- **Webデモ＝取り込み層**：貼る、分ける、根拠を確認する、可視化する、出力する
- **ローカルKX＝実働中の精錬層**：開発者自身が使っている、人間確認型Markdown運用
- **配布Skill＝今後の提供層**：一般利用者へのインストールと保存アダプター

ObsidianとNotionには認証が必要で、Skill版のロードマップに含まれると説明する。動くように見える偽ボタンは置かない。

応募文では次を正直に説明する。

- 仮想的な用途ではなく、開発者自身のKX実運用から生まれた
- Webデモは判断を取り込む段階を、誰でも理解・体験できる形にしたもの
- Markdownで持ち運べるため、プロジェクトやAIツールをまたぐ継続性を作れる
- 将来のAI作業を本人の判断軸へ寄せる助けにはなるが、本人を完全再現すると保証しない

## 10. テストと受入条件

### 根拠とAPI契約

- 日本語、絵文字、サロゲートペア、CRLF改行で位置が正しく往復する
- 1文字ずれ、変更された抜粋、異なるハッシュを拒否
- 0件・6件のテーマ、重複ID、範囲外、過剰な重複、過剰な全文カバーを拒否
- 修正タイトルに命令を書いても指示として扱わない
- 既存の根拠・推論サニタイズと高影響判断の注意表示をテーマごとに維持

### 並列処理と部分成功

- 実測した同時リクエスト数が2を超えない
- 5件中1件がタイムアウトしても4件の結果を保持
- 再試行は失敗した1テーマだけ
- 二重クリックでリクエストが増えない
- 中止後も完成済み状態が壊れない

### 長文と安全性

- 30,000文字のASCII、日本語、絵文字を確認
- Content-Length偽装とストリーム超過を意味処理前に拒否
- 文字起こし、APIキー、プロバイダー詳細、モデル出力を公開エラー、console、クライアントバンドルに出さない
- レート制限と、AI内部再試行を含む最大呼び出し数を確認
- 単一テーマ、3テーマ、5テーマ、再試行ありの各ケースで推定料金を計算し、通常5テーマが$0.10未満、最悪系が概ね$0.20以内であることを確認

### サマリーとZIP

- サマリーの順番が常に一定で、失敗テーマを明示
- ZIPファイル名で`../`、重複、長すぎる名前、Unicode境界を安全に処理
- 全成功・一部成功でmanifestと2種類のMarkdownが正しく入る
- 全失敗時はZIPを生成しない

### ブラウザ受入

- PC 1440pxとスマホ375pxで、貼り付け、検出、確認、タイトル修正、選択、生成、部分失敗、再試行、結果移動、ZIP保存を確認
- 現在の単一判断モード、6項目、根拠表示、KX出力、プライバシー表示を壊さない
- 本番では2〜3テーマを生成する限定的な文字起こしを使い、リクエスト数、時間、結果、ZIP内容、情報露出を確認

## 11. 完了条件

次をすべて満たしたら、この段階を完了とする。

1. 最大30,000文字から、根拠位置を持つ1〜5テーマを検出できる
2. 利用者が生成前にテーマを選択・除外し、タイトルを修正できる
3. 最大2並列で生成し、一部失敗しても成功結果を保持できる
4. 会議サマリー、アクション一覧、安全なMarkdown ZIPを生成できる
5. Web版・ローカルKX・将来Skill版の関係を画面と応募文で正確に説明できる
6. 保存や秘密情報漏えいなしで、自動テストと本番受入を通過する
